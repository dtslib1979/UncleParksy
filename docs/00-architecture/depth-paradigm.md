## 1. 사고 흐름 복기: 어디서 시작해서 여기까지 왔나

### A. LLM 노가리 단계
- 그냥 대화하다가 LLM이 코드를 만들어주는 걸 발견
- 즉시 결과물(웹페이지)이 떨어지는 재미로 Tistory에 업로드하며 놀기 시작

### B. Tistory 단계
- 웹페이지 코딩을 LLM에게 맡기면 "콘텐츠 공장" 가능하다고 판단
- 그런데 Tistory는 자바스크립트 제약 때문에 원하는 인터랙션이 안 나옴

### C. GitHub Pages 단계
- 제한을 피하려고 GitHub Pages로 이동
- Copilot Chat + ChatGPT 인스트럭션 기반으로 복붙 개발 루프 확립
- 이 시기에는 "페이지 단위 생산"이 분명 의미 있었음

### D. 출판사(MDX) 단계
- 단발 페이지가 아니라 MDX 기반 '출판사' 형식으로 구조를 통일
- 레포 여러 개, 도메인 3개로 나눠서 트릴로지 운영
- 비즈니스 도메인(.com)은 워드프레스로 보험까지 깔아둠

### E. Claude Code 한 달 이후
- **사고의 전환점**
- 프로그래밍을 "완성 제품 엔드포인트"로 못 잡아도 됨
- 대신 서식/템플릿/파이프라인 자체를 엔드포인트로 잡으면 됨
- 수익 엔드포인트는 결국 **YouTube로 확정**

### F. 비용/운영 현실화
- 안 쓰는 도메인 유지비가 고정비로 체감
- 결론:
  - YouTube가 못하는 것만 웹에서 한다
  - 웹은 "미디어 자동화/출판 공장" 역할
  - 출판→방송(YouTube)까지 이어지는 파이프라인 구축

## 2. 핵심 전환: YouTube ↔ Program 루프

```
1. YouTube에서 개념/사고를 말한다
        ↓
2. 웹에서 그 사고를 프로그램(서식)으로 만든다
        ↓
3. 사용자가 그 프로그램을 체험한다
        ↓
4. YouTube는 그 체험을 "체험 영상"으로 다시 보여준다
        ↓
5. 다시 웹에서 더 고도화된 프로그램이 나온다
        ↓
      (무한 순환)
```

**핵심 통찰:**
- 기존처럼 "영상 → 글"이 아니라
- **영상 ↔ 프로그램이 서로를 생성하는 출판 구조**

박씨는 "영상 제작자"가 아니라
**생각을 '작동하는 구조'로 바꿔 출판하는 사람**이 된다.

---

## 3. 프로그램의 변수화: 복붙으로 자기화

### 원칙
- 프로그램이 고정 콘텐츠가 아니라
- 유저가 복사-붙여넣기로 자기 것이 되게 만든다
- 변수/입력칸에 유저가 재료를 넣으면 결과가 "그 사람 버전"으로 바뀜

### 결과물 형태
- 소스는 YouTube 영상의 링크/오픈소스로 공개 가능
- 웹에서 돌아가는 체험 프로그램으로 제공

### 핵심 전환
> "보는 콘텐츠"가 아니라
> **가져가서 바꾸는 콘텐츠**

## 4. 공리: "모두가 손 안에 LLM을 갖고 있다"

### 공리 선언
> 모든 사람은 이미 손 안에 LLM이 있다.
> LLM에게 물어 답을 얻고, 그 결과를 여기다 붙여 넣으면
> 자기가 원하는 콘텐츠/세계/프로그램을 경험할 수 있다.

### 시스템의 역할
- 우리는 "정답"을 제공하는 게 아니라
- **LLM 출력물을 '체험 가능한 구조'로 변환하는 서식/프로그램을 제공**한다

### 확장: 체험 → 제작까지
페이지 안에서 자동으로 따라붙는 제안:
- "이 결과물을 HTML로 저장해라"
- "GitHub Pages로 배포해라"
- "PWA로 만들어라"
- "APK로 감싸라"
- "이 화면 그대로 영상 찍어 유튜브로 가라"

### 최종 흐름
```
LLM 답변 → (복붙) → 프로그램 체험 → 제작/배포 제안 → 영상화
```

이게 **출판-방송-자동화 삼각형**.

## 5. 가드레일: 페르소나 × KR 메리트

### 페르소나 시스템
사고 모드 선택기:
- 화가 박씨
- 블로거 박씨
- 철학자 박씨
- 뮤지션 박씨
- 기능인 박씨

### KR 메리트
페르소나에 곱해지는 소스:
- **Bluff** (허세)
- **Halfblood** (반쪽 언어)
- **Aggro** (어그로)
- **Shaman** (무속)

### 변환 공식
```
LLM 결과물 × (박씨 페르소나) × (KR 메리트) → 체험 가능한 프로그램/서식
```

### 이 조합이 작동하는 이유
- 유저는 막막하지 않다 (가드레일)
- 하지만 답을 강요받지도 않는다 (LLM이 재료)
- 결과는 항상 "내 버전"이 된다 (복붙/변수화)
- 시스템은 흔들리지 않는다 (페르소나×메리트가 톤/방향 고정)

## 6. 수직 지층 구조: Depth System

### 핵심 전환
모바일이라는 제약을 "단점"으로 처리하지 말고,
**'깊이'라는 미학으로 전환**한다.

### 스크롤의 재정의
- 스크롤이 짜증 나는 이유: "페이지가 많아서"가 아니라 **내가 어디에 있는지 모르기 때문**
- 해결: 화면 위에 항상 좌표 오버레이
  ```
  [Depth 3 / Laboratory]
  You are inside: Parksy Engine
  ```

### 지층 구조
```
표면 (YouTube에서 온 사람)
 └─ Depth 0: Entrance
     └─ Depth 1: Persona 선택
         └─ Depth 2: Merit 모드
             └─ Depth 3: 실험실 (복붙)
                 └─ Depth 4: 출력/배포
                     └─ Depth 5: 기록/아카이브
                         └─ Depth 6: Core (사고 엔진)
```

### UX 원칙
- 좌우 이동 없음
- 탭 최소화
- 뒤로 가기 = 위로 올라감
- 앞으로 = 더 깊이
- 항상 상단에 HUD 오버레이

### 이 구조의 강점
1. 모바일에 완벽히 맞는다
2. "페이지가 많다"는 느낌이 없다
3. 유저가 모험 중이라는 감각을 가진다
4. CAVE 철학(동굴, 밀실, 핵, 맨틀, 내면)과 100% 정합
5. YouTube에서 말한 걸 "자, 이제 내려와 봐"로 연결

### 정체성
- 블로그 구조 ❌
- 코스 구조 ❌
- 문서 구조 ❌
- **"사고를 탐험하는 지형"** ✓

## 7. 선언문

> YouTube는 입구, GitHub Pages는 세계, 프로그램은 생각의 서식이다.
> 그리고 유저는 LLM으로 만든 재료를 복붙해서 자기 세계를 체험하고,
> 그걸 저장/배포/제작까지 이어간다.

## 9. 남은 리스크

### A. 7층은 과욕
MVP는 4층까지만:
```
Depth 0: Entrance
Depth 1: 선택 (Persona + Merit 합침)
Depth 2: Laboratory
Depth 3: Output/Archive
```
Core는 나중에 열어도 됨.

### B. 스크롤 성능
해결책:
- 각 층은 `100vh` 스냅
- 이펙트는 현재 층에서만 활성화
- Intersection Observer로 층 감지

### C. "올라가기" UX
해결책:
- 상단 HUD에 "↑ Surface" 버튼
- 스크롤 올리면 자동으로 이전 층 스냅

## 11. 최종 정의

> **출판 = 실행 가능한 사고 (Executable Thought)**
> **방송 = 그 실행을 보여주는 쇼케이스**

박씨는 "설명하는 사람"이 아니라
**사고를 실행 파일로 배포하는 출판자**로 간다.
