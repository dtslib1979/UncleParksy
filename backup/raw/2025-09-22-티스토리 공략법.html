티스토리 공략법

중년작가지망생 2025. 9. 22. 19:35
티스토리 5축 워크플로우: 문답 아카이브 × 백서 요약
티스토리 5축 워크플로우 · 문답 아카이브

HTML(HTML) · 짭깃허브(GitHub Pages 대용) · 짭노션(Notion 대용) · 짭ChatGPT 커넥터(connector) · Second Brain(second brain) · 대화로그 아카이브 · 한국어 교재 플랫폼

복사 버튼이 막힐 경우(티스토리 JavaScript(JavaScript) 제한): 본문을 길게 눌러 전체 선택 → 복사.

Q1 “짭” 개념으로 티스토리를 핵심 파이프라인에 넣어도 되는가?

질문

정식 도구가 아닌 대용(짭)으로도, 실전 파이프라인의 뼈대를 세울 수 있는가?

  • 의도: GitHub Pages(GitHub Pages)·Notion(Notion)·API(API) 없이도, HTML(HTML) 발행/분류/백업이 가능한가.
  • 맥락: JS(JavaScript) 제약이 있는 티스토리지만, 즉시 배포·카테고리·AdSense(AdSense) 등 실전성이 있음.

답변/합의

가능. 티스토리는 **짭깃허브**(HTML 발행 + 카테고리)로, **짭노션**(리포트 HTML 저장 + 무료 URL)로, **짭ChatGPT 커넥터**(공개 URL 읽기 레퍼런스)로 작동한다. 핵심은 “읽기 중심, 즉시 발행, 카테고리=목차”다.

Q2 왜 Markdown 대신 HTML 인가?

질문

ChatGPT·Claude가 바로 HTML 코드를 생성하는데, Markdown으로 중간 변환할 이유가 있는가?

답변/합의

  • 굳이 없음. HTML은 시각·인터랙션(정책 허용 범위)을 즉시 반영한다.
  • 티스토리는 HTML 모드 지원. 네이버는 HTML 모드 불가 → 동적 표현 제한.
  • 결론: **직접 HTML 제작 → 티스토리에 즉시 발행**이 연속성·가독성에 유리.

참고: JS 전면 허용은 아님(플랫폼 정책/보안). CI/CD(CI/CD) 없이 “즉시 게시”가 강점.

Q3 “짭깃허브”로 불러도 되나? (JS 제한 vs 실전성)

질문

GitHub Pages처럼 JS 풀 스택을 못 쓰는데도, 깃허브 대용으로 명명 가능한가?

답변/합의

  • 가능. 이름은 “짭”이지만 목적은 **HTML 발행·분류·수익화**.
  • 장점: 즉시게시, 카테고리=교재 인덱스, 애드센스, 검색 유입(SEO(SEO)) 잠재력.
  • 보완: RSS(RSS)/API(API)로 GitHub 백업·미러링(반자동) 설계.
Q4 카테고리를 “교재 목차(Outline)”로 쓰는 방식

질문

HTML 포스트를 카테고리로 묶어 **책/교재**처럼 보여줄 수 있는가?

답변/합의

  • 가능. 카테고리를 챕터·섹션처럼 설계 → 블로그=교재형 인덱스.
  • 개별 포스트=단원/레슨. 링크=목차 간 연결.
  • 즉시 게시로 “연재형 교재” 운영이 쉬움.
Q5 수익화 경로: 검색 유입 × AdSense

질문

네이버만큼의 노출은 아니어도 실질 수익화가 가능한가?

답변/합의

  • 가능. 노출이 붙으면 페이지 열람→AdSense로 소액 수익.
  • 교재형 콘텐츠는 체류시간·회독률을 올려 광고 효율을 보강.
  • 긴 호흡의 “백서·강의노트”는 구독 동선 확보에 유리.
Q6 GitHub와의 연동: RSS/API로 백업·관리

질문

티스토리와 GitHub를 어떻게 연결할까?

답변/합의

  • RSS를 주기적으로 긁어 HTML 아카이브 저장.
  • API 또는 수동 스크립트로 포스트 백업/버전관리.
  • 역방향: GitHub → 티스토리 발행도 가능(반자동).

핵심: 프론트(티스토리) / 백엔드(GitHub) 이원화.

Q7 “짭노션”: 메모 소환이 아니라 HTML 리포트 저장소

질문

Notion처럼 노트를 쌓는 게 아니라, HTML 리포트/아이디어를 바로 저장·발행하는 모델이 유효한가?

답변/합의

  • 유효. ChatGPT/Claude가 HTML로 즉시 출력 → 티스토리에 붙여넣기.
  • 무료 퍼블릭 URL 자동 생성 → 배포/공유 용이.
  • 카테고리=목차, URL=레퍼런스. Notion 권한/로그인 장벽 회피.
Q8 “짭ChatGPT 커넥터”: 읽기 전용 Second Brain

질문

Notion/GitHub를 직접 커넥터로 붙이면 느리고 토큰 낭비. 대안?

답변/합의

  • 티스토리 공개 URL 목록을 ChatGPT Projects/Claude 프로젝트 지침에 등록.
  • 에이전트는 DB 쓰기 대신 URL 읽기로 참조(Reference)만 수행.
  • 쓰기 실패/지연 리스크 제거, 비용 절감, 속도 향상.

핵심: 쓰기(Write) 욕심을 버리고, 레퍼런스(Read)로 최적화.

Q9 진짜 자산은 리포트가 아니라 “대화 로그”다

질문

프롬프트만 주면 리포트는 재생산 가능. 무엇을 자산화할 것인가?

답변/합의

  • 대화 로그 자체. 사고과정/수정/반복의 흐름.
  • 모국어(한국어) 대화가 속도·정확도에서 우위.
  • 정리된 HTML 회의록으로 아카이브 → 티스토리에 게시.
Q10 아카이브 운영: 정리→HTML→게시의 루틴화

질문

대화 로그를 어떻게 꾸준히 적산(積算)할까?

  • 세션 종료→핵심 문답 정리→HTML 변환→티스토리 게시.
  • 카테고리로 맥락 분류(교재형 인덱스).
  • 주기적 GitHub 미러링으로 장기 백업.
Q11 한국어 교재 플랫폼으로의 확장

질문

티스토리는 GitHub도 네이버도 아닌 중간 지점. 글로벌 한국어 학습 플랫폼 가능?

답변/합의

  • 전 세계 사용자 ID 개방. 퍼블릭 URL 배포 용이.
  • 상식적·평이한 문체의 필터링 자료=한국어 읽기 교재로 최적.
  • 카테고리 기반 커리큘럼 구성 → 반복 노출로 학습 효율 향상.
Q12 최종 구조: “5축 티스토리 워크플로우”
  • ① 짭깃허브: HTML 발행 + 카테고리 인덱스 + AdSense + RSS/API 연동.
  • ② 짭노션: HTML 리포트 저장 + 무료 퍼블릭 URL.
  • ③ 짭ChatGPT 커넥터: 공개 URL 읽기 전용 참조 = second brain.
  • ④ 대화로그 아카이브: 회의록 중심 자산화(한국어 우선).
  • ⑤ 한국어 교재 플랫폼: 글로벌 대상 읽기 교재 허브.

명명은 “짭”이지만, 기능적 목적은 **실전 출판/학습 운영체계**다.

  • 카테고리를 목차(Outline)로 고정하면 연재와 교재가 동시에 가능해지는가?

  • 읽기 전용(Reference) 최적화가 쓰기(Write) 자동화의 미련을 지울 만큼 효율적인가?

  • 대화 로그를 “회의록 데이터베이스”로 표준화하면 창작 속도가 가속되는가?
  • 티스토리의 애매한 포지션이 오히려 “한국어 읽기 교재” 틈새에서 경쟁력이 되는가?
  • RSS(RSS)/API(API) 반자동 백업이 장기 보존성에서 어떤 임계점을 만들어내는가?

백서 요약

취지/목적

  • 티스토리를 HTML 중심의 라이트 출판·학습 OS(OS)로 전환.
  • 정식 커넥터 대신 공개 URL 참조로 second brain 구현.
  • 대화 로그를 자산화하여 교재/연재/수익화로 연결.

문답 요약 리스트

  • 짭 전략 유효: 읽기 중심 + 즉시 발행 + 카테고리 인덱스.
  • Markdown 대신 HTML: 시각·동적 표현·연동에 유리.
  • 짭깃허브: JS 제한 대신 운영 민첩성과 수익화.
  • 짭노션: HTML 리포트 저장 + 무료 퍼블릭 URL.
  • 짭ChatGPT 커넥터: URL 리스트로 빠른 레퍼런스.
  • 대화로그 아카이브: 유일 자산은 사고흐름.
  • 한국어 교재 플랫폼: 글로벌 독자에게 읽기 자료 허브.
